# 文件发布到公共知识空间 · 业务域一致性校验 PRD

- 文档日期：2026-06-24
- 状态：待评审
- 关联分支：fix/file-detail-tags（实现另起分支）

## 1. 背景与目标

文件发布到公共知识空间时，必须保证「文件所属业务域」与「目标公共知识空间所属业务域」一致；不一致则禁止发布并给出明确提示，避免文件被发布到错误的业务域空间，污染公共知识体系。

- **发布动作**最终由上游 BiSheng 完成，但**业务域一致性校验由门户执行**：门户先校验，校验通过后再调用 BiSheng 发布接口；不通过则阻断，不调用 BiSheng。
- 业务域配置（编码、绑定空间）均在门户后台维护，因此门户是校验逻辑与配置的唯一权威方。

### 目标
1. 发布前自动校验业务域一致性，不一致一律阻断并提示。
2. 让「公共知识空间 → 业务域编码」的映射来源清晰、可配置、可覆盖全部公共空间。

### 非目标（本期不做）
- 不改造后台「知识空间」配置栏（见 §7 排查结论，本期保持现状）。
- 不在门户新建文件上传/发布的完整 UI（沿用既有发布入口；门户仅在发布编排链路中插入校验）。
- 不做业务域历史脏数据（一个空间被多业务域绑定）的批量清洗。

## 2. 名词与数据模型

| 名词 | 说明 | 代码锚点 |
|---|---|---|
| 业务域 DomainConfig | 有 `name`、`code`（业务域编码）、绑定一个知识空间、`enabled` 等 | `backend/app/schemas/portal_config.py` `DomainConfig` |
| 业务域编码 code | 对应文件编码第 3 段（如 `SGGF-STD-PP-…` 中的 `PP`），统一转大写比较 | 同上，`DomainConfig.code` |
| 公共知识空间 | 知识空间配置中 `space_level == "public"` 的空间 | `SpaceConfig`，`space_level` |
| 文件编码 | 格式 `SGGF-STD-{CODE}-{SEQ}`，第 3 段为业务域编码 | `backend/app/services/domain_file_count_service.py` 解析逻辑 |
| 绑定关系 | 业务域 → 知识空间（DomainConfig 绑定一个 space_id）；一个空间可被多个业务域绑定 | `DomainConfig.space_ids` |

### 关键映射
- **文件业务域 code** = 文件编码第 3 段（解析自文件编码，复用 `domain_file_count_service` 的解析）。
- **公共空间业务域 code 集合** = 所有「绑定空间 == 目标空间」且 `enabled` 的业务域的 `code` 集合（去空、trim、转大写）。
  - 因存量数据中一个空间可被多个业务域绑定（如「财务」空间被「财务」「生产」两个业务域绑定），该集合可能为多值。

## 3. 校验规则（核心）

设目标空间为 `target_space_id`（必为公共空间），待发布文件编码为 `file_encoding`。

```
1. file_code = parse_domain_code(file_encoding)   # 取编码第3段，trim+大写
   - 若 file_code 为空 / 编码缺失 / 格式非法 → 阻断（reason: FILE_CODE_MISSING）

2. space_codes = { d.code.strip().upper()
                   for d in domains
                   if d.enabled
                   and d.code.strip()
                   and target_space_id in d.binding_space_ids }
   - 若 space_codes 为空（无业务域绑定该空间，或绑定的业务域均无 code）
     → 阻断（reason: SPACE_DOMAIN_UNCONFIGURED）

3. if file_code in space_codes:  → 允许发布（命中任一即一致）
   else:                          → 阻断（reason: DOMAIN_MISMATCH）
```

判定原则汇总：
- **一致性**：文件 code 命中目标空间 code 集合中**任一**即视为一致（多域绑定容忍现状）。
- **大小写/空格**：统一 `strip().upper()` 后比较。
- **缺失即拒**：文件 code 无法解析、或目标空间未配置业务域，**一律禁止发布**（不放行、不静默跳过）。

## 4. 校验流程（时序）

```
用户在发布入口选择「发布到公共知识空间」并指定目标公共空间
        │
        ▼
门户发布编排接口  POST /api/v1/knowledge/publish
        │
        ├─ 1. 解析文件编码 → file_code
        ├─ 2. 反查目标空间业务域 code 集合
        ├─ 3. 一致性判定
        │
   ┌────┴─────────────────────────┐
   │ 一致                          │ 不一致 / code缺失 / 未配置
   ▼                              ▼
调用 BiSheng 发布接口            返回 4xx + 结构化原因 + 文案
   │                              （不调用 BiSheng）
   ▼                              ▼
发布成功                        前端弹窗提示，阻断发布
```

可选：提供**预校验接口** `POST /api/v1/knowledge/publish/precheck`，供前端在用户选择目标空间后即时提示/禁用发布按钮（与正式发布共用同一校验逻辑）。

## 5. 边界场景

| 场景 | 处理 | reason_code |
|---|---|---|
| 文件编码缺失 / 第3段为空 / 格式非法 | 阻断，提示补全编码 | `FILE_CODE_MISSING` |
| 目标空间未被任何业务域绑定 | 阻断，提示联系管理员配置业务域 | `SPACE_DOMAIN_UNCONFIGURED` |
| 目标空间绑定的业务域均无 code | 阻断，同上 | `SPACE_DOMAIN_UNCONFIGURED` |
| 目标空间被多个业务域绑定 | 收集全部 code，文件 code 命中任一即放行 | —（正常一致逻辑） |
| 文件 code 与空间 code 集合无交集 | 阻断，提示业务域不一致 | `DOMAIN_MISMATCH` |
| 目标空间非公共空间 | 本特性仅约束发布到公共空间；非公共目标不在本期范围 | N/A |

## 6. 后台配置改造：业务域「绑定空间」候选范围

**现状**：业务域编辑弹窗「绑定空间」下拉候选来自 `config.spaces` 全量（含个人/团队/部门/公共各级别）。

**改造**：候选范围改为**所有公共知识空间**，即知识空间配置中 `space_level == "public"` 的空间。

- 数据来源：知识空间配置（`SpaceConfig`，`space_level=="public"`）。
- 目的：保证每个公共知识空间都可被绑定业务域，从而其业务域编码完整、可被发布校验反查到。
- 影响面：仅改 `DomainEditorDialog` 绑定空间下拉的候选过滤（`frontend/src/pages/AdminPage.tsx`），不改业务域数据结构、不改保存校验主流程；保存时仍校验所选 space 存在且为公共空间。
- 兼容：存量业务域若已绑定非公共空间，编辑时给出提示并要求改绑公共空间（或在保存校验中拦截）。

## 7. 知识空间配置栏排查结论（本期不动）

针对「后台『知识空间』配置栏是否还有用、能否删除」的排查结论：**仍在使用，不可删除**。`SpaceConfig` 是门户可见性枢纽，至少驱动以下功能：

| 用途 | 代码锚点 | 删除后果 |
|---|---|---|
| 未登录用户可见范围白名单 | `knowledge_service.get_enabled_space_ids()` | 前台无法限定公开内容范围 |
| 首页「总文档数」统计 | `home-stats` 累加 `spaces.file_count` | 首页统计失效 |
| 业务域「绑定空间」候选 | 编辑弹窗 `spaces` 来自 `config.spaces` | 无可绑定项 |
| 问答未登录公共空间树 | `list_public_config_spaces()`（`space_level=="public"`） | 问答选库不可用 |
| 前台按 space_level 过滤搜索 | `get_enabled_space_ids_by_level()` | 公共/部门/团队筛选失效 |

**结论**：本期保留知识空间配置栏，仅记录排查结果，不做删除/迁移。`tag_count` 字段从未被读写，可在后续清理时安全移除（非本期范围）。

## 8. 接口设计（门户侧）

### 8.1 校验服务（内聚逻辑）
新增 `DomainConsistencyService`（或并入现有 knowledge 服务），核心方法：

```
validate(file_encoding: str, target_space_id: int) -> ConsistencyResult
```

`ConsistencyResult`：
```json
{
  "allowed": false,
  "reason_code": "DOMAIN_MISMATCH",
  "message": "该文件属于「轧线」业务域，与目标空间「能源」业务域不一致，不允许发布",
  "file_domain": { "code": "PP", "name": "轧线" },
  "space_domains": [ { "code": "EN", "name": "能源" } ]
}
```

- 复用：编码解析复用 `domain_file_count_service` 的 code 解析；业务域配置来自 `portal_config_service`。

### 8.2 发布编排接口
`POST /api/v1/knowledge/publish`
- 入参：文件标识（含编码）、`target_space_id`。
- 流程：先 `validate()`；`allowed==false` 返回 4xx + `ConsistencyResult`，不调用 BiSheng；`allowed==true` 调用 BiSheng 发布接口并返回结果。

### 8.3 预校验接口（可选）
`POST /api/v1/knowledge/publish/precheck`
- 入参同上，仅返回 `ConsistencyResult`，不触发发布。供前端即时提示/禁用按钮。

## 9. 提示文案（前端弹窗）

| reason_code | 文案 |
|---|---|
| `DOMAIN_MISMATCH` | 该文件属于「{文件域名称}」业务域，与目标空间「{空间域名称}」不一致，不允许发布。 |
| `FILE_CODE_MISSING` | 文件编码缺失或不规范，无法识别业务域，请先补全文件编码后再发布。 |
| `SPACE_DOMAIN_UNCONFIGURED` | 目标公共空间未配置业务域，请联系管理员在后台为该空间绑定业务域后再发布。 |

> 文案中的域名称取自业务域配置 `name`；`file_domain.name` 若 code 在配置中无对应业务域，则回退展示 code 本身。

## 10. 验收标准

1. 文件 code 与目标公共空间任一绑定业务域 code 一致 → 发布成功，BiSheng 收到发布请求。
2. 文件 code 与目标空间所有绑定业务域 code 均不一致 → 阻断，提示 `DOMAIN_MISMATCH`，BiSheng **未**收到发布请求。
3. 文件编码缺失/非法 → 阻断，提示 `FILE_CODE_MISSING`。
4. 目标空间无业务域绑定 / 绑定域无 code → 阻断，提示 `SPACE_DOMAIN_UNCONFIGURED`。
5. 多业务域绑定同一空间时，命中任一即放行。
6. 业务域编辑「绑定空间」下拉仅展示公共知识空间（`space_level=="public"`）。
7. 知识空间配置栏功能保持现状（首页统计、未登录范围、问答选库均正常）。

## 11. 开放项 / 待确认

1. **发布入口位置**：门户「发布到公共知识空间」的具体前端入口（页面/按钮）尚未在代码中定位，需实现前确认其位置与触发参数（文件编码、target_space_id 的取值来源）。
2. **批量发布**：若支持批量发布，建议「逐文件校验」——一致的放行、不一致的逐条列出原因阻断（部分成功）；是否支持批量及交互需产品确认。
3. **存量多域绑定清洗**：本期容忍多域绑定（命中任一即放行）；后续是否收敛为「一空间一业务域」需另行决策。
