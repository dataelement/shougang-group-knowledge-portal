# PRD — 业务域文件编码配置 与 首页知识数量改造

> 状态：待评审 · 创建日期：2026-06-03 · 适用仓库：shougang-group-knowledge-portal（门户）
> 关联：依赖 BiSheng 侧**新增一个聚合统计接口**（见 §5）；门户侧新增三级缓存。

---

## 1. 背景与现状

- 后台「业务域管理」中，一个业务域（`DomainConfig`）目前包含名称、绑定空间（`space_ids`）、图标、颜色、背景图等字段，**没有编码字段**（[portal_config.py:27](../backend/app/schemas/portal_config.py#L27)、[adminDomains.ts](../frontend/src/utils/adminDomains.ts)）。
- 首页「业务域导航」卡片上的「知识数量」当前取值逻辑：**累加该业务域绑定空间的 `file_count`**（[HomePage.tsx:470-476](../frontend/src/pages/HomePage.tsx#L470-L476)）。即数量口径绑定在「空间」上，与文件本身的业务属性无关。
- 文件其实已自带编码：门户从 BiSheng 返回项里通过 `FILE_ENCODING_KEYS` 提取 `file_encoding`（[knowledge_service.py:76-89](../backend/app/services/knowledge_service.py#L76-L89)），并在检索中只保留解析成功（`status == SUCCESS_STATUS == 2`）的文档文件（[knowledge_service.py:1002-1015](../backend/app/services/knowledge_service.py#L1002-L1015)）。
- 文件编码格式示例：`SGGF-STD-PP-20260500000001`，以 `-` 分四段：
  1. 第 1 段 `SGGF`：集团/来源前缀
  2. 第 2 段 `STD`：**文档类型编码**
  3. 第 3 段 `PP`：**业务域编码** ← 本需求关注
  4. 第 4 段 `20260500000001`：流水号
- 编码由上游（大模型分类器）生成，规则见 [附录 A](#附录-a编码生成规则参考仅作背景)。**本次门户开发不负责生成编码**，仅消费已有的 `file_encoding`。

---

## 2. 目标与范围

### 2.1 目标
1. **业务域支持配置「业务域编码」**（编码第 3 段，如 `PP`）：在后台「业务域管理」编辑界面新增编码字段，允许自由填写并提供 14 个枚举值作为快捷候选。
2. **首页业务域卡片「知识数量」改为编码口径**：数量 = 在 **BiSheng 全部知识库** 中，`file_encoding` 第 3 段等于该业务域编码、且**解析成功**的文档文件数。
3. 因编码第 3 段匹配**无法走左前缀索引**、大数据量下实时查询有性能问题，门户侧引入**三级缓存**（内存 → SQLite → BiSheng），并采用 **stale-while-revalidate**（返回旧值 + 后台刷新）策略。

### 2.2 明确不做（范围外）
- **不实现编码生成**：不接大模型、不对文件自动打码、不回写编码。文件编码假定由上游/BiSheng 已生成（附录 A 仅作背景说明）。
- **不配置文档类型编码**：本次后台仅新增「业务域编码」（第 3 段）一个字段，不维护文档类型（第 2 段）枚举。
- **不改业务域↔空间绑定关系**：`space_ids` 绑定保留（业务域详情/跳转仍用），仅**不再用于卡片数量统计**。
- 不改首页其它统计（顶部「N 篇文档」等口径保持现状）。

## 3. 名词

| 术语 | 含义 |
|---|---|
| 业务域编码 | 文件编码第 3 段，如 `PP`（生产）、`QM`（质量）。一个业务域配一个。 |
| 解析成功 | BiSheng 文件 `status == 2`（`SUCCESS_STATUS`），且为文档类型文件（`file_type == FILE_TYPE`）。 |
| 全部知识库 | BiSheng 管理的所有知识库，**与门户启用库名单、登录态均无关**（见 §5 口径）。 |
| 三级缓存 | 内存缓存 → SQLite 持久化缓存 → BiSheng 聚合接口，逐级回源。 |

## 4. 需求详述与验收标准

### R1 业务域新增「业务域编码」配置字段
- 在 `DomainConfig` 增加字段 `code: str = ""`（业务域编码，编码第 3 段）。
- 后台「业务域管理」编辑/新增弹窗新增「编码」输入项：
  - **可编辑输入框 + 14 个枚举值候选**（combobox / `datalist`），方便快速填写；同时**允许管理员自由输入**，以应对后续编码规则变更。
  - 候选枚举（业务域编码）：`PP 生产`、`QM 质量`、`PM 设备`、`EM 能源`、`SA 安全`、`EN 环保`、`IM 投资`、`RD 研发`、`MM 采购`、`SD 营销`、`FI 财务`、`HR 人力`、`IT 信息`、`AD 管理`。
  - 保存时归一化：`trim` 去空白并**转大写**（如输入 `pp` 存为 `PP`）。
  - **允许留空**：未配编码的业务域，其卡片「知识数量」按 0 处理（见 R3）。
  - **不强制唯一**：允许两个业务域配同一编码（不报错，各自按该编码统计，数值会一致）。
- 验收：
  - 编辑业务域可设置/修改/清空编码并保存；刷新后回显正确（大写归一）。
  - 候选下拉能快速填入 14 个枚举之一，也能手输非枚举值并保存成功。

### R2 编码下发前端（公共配置）
- `code` 字段随现有公共配置链路原样下发到前端（与 `domains` 其它字段同路径，无需新增下发接口）。
- 验收：前端拿到的每个 domain 含 `code` 字段。

### R3 首页业务域卡片「知识数量」改为编码口径
- 卡片「知识数量」取值由「累加绑定空间 `file_count`」改为「该业务域 `code` 对应的编码匹配数量」。
- 取数来自门户新增的**业务域计数接口**（见 §5），按 `code → count` 映射；前端用 `domain.code` 取数。
- 边界：
  - 业务域 `code` 为空 → 显示 `0`。
  - 计数接口失败/超时 → 显示 `0`（**非阻断**，首页其余内容照常渲染）。
  - 多个业务域共用同一 `code` → 各自显示相同数值（预期）。
- 验收：
  - 配好编码的业务域，卡片数量 = 全部知识库中该编码且解析成功的文档数（与 §5 口径一致）。
  - 未配编码 / 接口异常时显示 0，页面不报错。

### R4 三级缓存与 stale-while-revalidate
- 门户「业务域计数接口」按编码缓存，逐级回源：
  1. **内存缓存**（进程内，带 `fetched_at`）。
  2. **SQLite 持久化缓存**（进程重启后仍可用，避免冷启动打满 BiSheng）。
  3. **BiSheng 聚合接口**（最终数据源，见 §5）。
- 过期/未命中策略为 **stale-while-revalidate**：
  - 命中（未过期）→ 直接返回。
  - 命中但**已过期** → **先返回旧值**，同时**后台异步**回源刷新（内存 + SQLite），下次请求生效；后台刷新失败则保留旧值。
  - **完全无缓存**（首次冷启动且 SQLite 也无）→ 触发一次回源；在结果落库前对应编码返回 `0`（前端显示 0），刷新完成后下次请求即生效。
- 后台刷新需**防并发重复回源**（同一时刻对同一批编码只发起一次刷新，如用刷新锁 / in-flight 标记）。
- 缓存键 = 业务域编码（`code`）。回源时取**当前已配置且非空**的全部业务域编码，一次性批量查询。
- 验收：
  - 连续多次访问首页，BiSheng 聚合接口在 TTL 窗口内**最多被回源一次**（可通过日志/抓包验证）。
  - 数据变化后，最长在 TTL + 一次访问触发的刷新周期内反映到首页。
  - 后台刷新失败时，首页仍展示上次的缓存值，不报错。

### R5 缓存 TTL 后台可配
- 新增站点级配置项 `domain_count_cache_ttl_seconds`（缓存有效期，单位秒），**默认 43200（12 小时）**。
- 在后台「站点配置」页提供该项编辑（数值校验：正整数；建议下限如 60 秒，避免误填 0 导致频繁回源）。
- TTL 变更即时影响后续缓存命中判定（无需重启）。
- 验收：管理员把 TTL 改小后，过期判定按新值生效；改回 12h 亦生效。

## 5. BiSheng 聚合统计接口（需 BiSheng 新增）

> 这是本需求**唯一**需要 BiSheng 侧改动的点。门户侧封装一层缓存代理，不直接把 BiSheng 接口暴露给前端。

### 5.1 接口职责
给定一组业务域编码，返回每个编码在 **BiSheng 全部知识库** 中匹配的文件数。

### 5.2 统计口径（关键，需 BiSheng 严格遵守）
对 BiSheng 全部知识库中的文档文件，计数满足以下**全部**条件者：
1. 文件**解析成功**（`status == 2`）；
2. 文件为文档类型（与门户检索一致的 `file_type`）；
3. 文件 `file_encoding` 以 `-` 分割后**第 3 段**（index 2）**等于**给定业务域编码（建议大小写不敏感比较，或约定均为大写）。
- 范围为 **BiSheng 管理的全部知识库**，**忽略门户的「启用库」名单，与登录态无关** —— 即任何用户在首页看到的卡片数量一致。
- 第 3 段不存在 / 编码格式不规范的文件，不计入任何编码。

### 5.3 建议契约（最终以双方联调为准）
- 请求：`POST /api/v1/knowledge/shougang-portal/domain-file-counts`
  ```json
  { "codes": ["PP", "QM", "PM"] }
  ```
- 响应：
  ```json
  { "status_code": 200, "data": { "counts": { "PP": 12, "QM": 3, "PM": 0 } } }
  ```
- 未匹配到的编码返回 0（或缺省，由门户侧补 0）。

> 性能说明：编码第 3 段匹配无法走左前缀索引，BiSheng 侧实现可考虑预聚合/物化（如按编码段建独立字段或维护计数表），具体由 BiSheng 团队评估；门户侧三级缓存进一步降低回源频率。

## 6. 门户侧接口与数据流

### 6.1 新增门户接口
- `GET /api/v1/knowledge/domain-file-counts`
  - 无需登录态（口径与登录无关）。
  - 行为：读取已配置业务域的非空 `code` 集合 → 走三级缓存（§R4）→ 返回 `{ "counts": { "<code>": <int> } }`。
  - 前端首页独立调用该接口填充卡片数量，**不阻塞**首页其它内容渲染。

### 6.2 SQLite 缓存存储
- 复用 `SQLiteConfigStore` 的「单行 JSON 文档」模式（[config_store.py](../backend/app/services/config_store.py)）：
  - 新增允许表名 `domain_count_cache`（加入 `_ALLOWED_TABLES` 白名单）。
  - 文档结构（单行）示例：
    ```json
    { "PP": { "count": 12, "fetched_at": "2026-06-03T08:00:00Z" },
      "QM": { "count": 3,  "fetched_at": "2026-06-03T08:00:00Z" } }
    ```
  - 过期判定：`now - fetched_at > ttl`。
- 内存缓存为进程内同结构映射，读优先于 SQLite；回源成功后同时写回内存 + SQLite。

### 6.3 数据流（首页一次访问）
```
前端首页 → GET /knowledge/domain-file-counts
           ↓
      内存命中未过期? ──是──→ 返回
           ↓否
      SQLite 命中未过期? ──是──→ 回填内存 → 返回
           ↓否（无缓存或已过期）
      返回现有旧值（无则 0） + 后台异步回源
                              ↓
                  BiSheng 聚合接口（全部知识库按编码统计）
                              ↓
                  写回内存 + SQLite（下次访问生效）
```

## 7. 改动点（实现指引）

> 门户前后端 + BiSheng 一处新增接口；不改业务域↔空间绑定语义。

1. **BiSheng（§5）**：新增按业务域编码统计全库解析成功文件数的聚合接口。

2. **后端 schema [portal_config.py](../backend/app/schemas/portal_config.py)**
   - `DomainConfig` 增 `code: str = ""`。
   - `SiteConfig`（或合适的站点配置块）增 `domain_count_cache_ttl_seconds: int = 43200`。

3. **后端服务**
   - `knowledge_service.py`：新增 `get_domain_file_counts()`，封装三级缓存 + 调 BiSheng 聚合接口 + 后台刷新（防并发）。
   - `config_store.py`：`_ALLOWED_TABLES` 增 `domain_count_cache`；缓存读写复用 `get_document` / `upsert_document`。
   - 新增门户路由 `GET /api/v1/knowledge/domain-file-counts`。
   - 后台「站点配置」更新逻辑纳入 `domain_count_cache_ttl_seconds`。

4. **前端后台 [adminDomains.ts](../frontend/src/utils/adminDomains.ts) + AdminPage 业务域编辑弹窗**
   - `DomainDraft`/`DomainConfig` 增 `code`；编辑表单新增「编码」输入（带 14 枚举候选、自由输入、保存转大写）。
   - 校验：允许空；非空时 trim+大写。
   - 「站点配置」页新增缓存 TTL 编辑项。

5. **前端首页 [HomePage.tsx](../frontend/src/pages/HomePage.tsx)**
   - 新增对 `GET /knowledge/domain-file-counts` 的调用，得到 `code → count`。
   - `domainTotals`（[L470-476](../frontend/src/pages/HomePage.tsx#L470-L476)）由「累加绑定空间 file_count」改为「按 `domain.code` 取计数」；空 code / 缺失 → 0。
   - 数量加载失败不阻断卡片渲染（显示 0）。

## 8. 边界与异常

- **业务域未配编码**：卡片数量 0；不影响业务域跳转（仍按绑定空间）。
- **BiSheng 聚合接口失败**：返回缓存旧值；无旧值则 0；后台刷新下次重试。
- **冷启动（内存 + SQLite 均无）**：首屏显示 0，后台刷新完成后下次访问显示真实值。
- **TTL 误配为 0 或负数**：按下限兜底（如 60s）或拒绝保存，避免频繁回源打垮 BiSheng。
- **多业务域共用编码**：数值一致，属预期。
- **编码大小写**：门户统一转大写存储；与 BiSheng 比较约定大小写不敏感或均大写，避免漏配。
- **编码格式异常的文件**（无第 3 段）：BiSheng 侧不计入，门户不特殊处理。

## 9. 决策记录（已确认）

1. **编码生成是否在范围** → 否。门户仅消费已有 `file_encoding`，不接大模型、不打码（附录 A 仅作背景）。
2. **统计范围与方式** → 新增 BiSheng 聚合接口按编码统计；范围为 **BiSheng 全部知识库，与启用库名单/登录态无关**。
3. **后台配置字段** → 仅「业务域编码」（第 3 段）；可自由填写 + 14 枚举候选；不维护文档类型编码。
4. **性能** → 门户侧三级缓存（内存 → SQLite → BiSheng）+ **stale-while-revalidate**（返旧值 + 后台刷新）。
5. **缓存 TTL** → 后台可配，默认 12 小时。

## 附录 A：编码生成规则（参考，仅作背景）

> 以下为上游大模型分类器生成 `file_encoding` 的规则，**本次门户开发不实现**，仅用于理解编码第 2 段（文档类型）/第 3 段（业务域）的含义。

输出格式：`文档类型编码-业务域编码`（如 `RPT-PP`）。

**文档类型（第 2 段）**：政策制度 `POL`、标准规范 `STD`、流程方法 `PRO`、技术规程与诀窍 `SPC`、报告 `RPT`、案例库 `CAS`、设计资产 `DGN`、专利与知识产权 `PAT`、培训资源 `TRN`、行业情报 `NEW`。

**业务域（第 3 段，本需求使用）**：生产 `PP`、质量 `QM`、设备 `PM`、能源 `EM`、安全 `SA`、环保 `EN`、投资 `IM`、研发 `RD`、采购 `MM`、营销 `SD`、财务 `FI`、人力 `HR`、信息 `IT`、管理 `AD`。
